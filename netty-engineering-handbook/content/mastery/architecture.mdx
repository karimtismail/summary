<StoryBlock title="Architecture Is Runtime Shape">
A network architecture is not a box diagram. It is a set of queues, protocols, ownership boundaries, and failure policies.
</StoryBlock>

## Intuition

The best networking architecture makes pressure visible and failure contained. It defines what each service promises on the wire and what happens when that promise cannot be kept.

## Real-world problem

A gateway accepts traffic, decodes custom messages, calls three services, and streams responses. Without explicit boundaries, one downstream slowdown fills gateway memory and degrades unrelated clients.

## Mental model

Design each boundary:

- protocol contract
- timeout and retry policy
- backpressure behavior
- memory ownership
- observability signals
- compatibility plan

The architecture is healthy when every boundary has a boring answer to four questions: **what can enter, how much can wait, how long may it take, and what happens when it fails?**

## Runtime behavior

Every async boundary introduces a queue. Every queue needs a maximum, a metric, and a policy. If the policy is not explicit, the default is usually memory growth.

<RuntimeTrace title="Boundary review checklist">
Ingress | Max connections, auth point, request size, idle timeout, overload response.
Protocol | Versioning, framing, compatibility, malformed input policy.
Work placement | EventLoop work, blocking work, CPU work, and executor limits.
Downstream | Timeout, retry budget, circuit behavior, and fallback policy.
Egress | Write watermarks, slow client policy, compression, and close semantics.
Operations | Metrics, logs, traces, dashboards, alerts, and runbooks.
</RuntimeTrace>

## Code example

```java
public record BoundaryPolicy(
    int maxFrameBytes,
    int maxQueuedMessages,
    int writeHighWatermarkBytes,
    long downstreamTimeoutMillis,
    boolean closeSlowClients
) {
    public static BoundaryPolicy productionDefault() {
        return new BoundaryPolicy(
            1 << 20,
            1_000,
            256 * 1024,
            500,
            true
        );
    }
}
```

This record is not a replacement for Netty config. It is a design artifact: it forces the team to name size, queue, pressure, timeout, and slow-client behavior before the code hides those choices.

<Warning title="Architecture without limits becomes memory policy">
If a design says "queue it", "retry it", or "broadcast it" without a cap, then the real policy is to spend memory until the process or downstream dependency pushes back.
</Warning>

## Production implications

Architecture reviews should ask how the system behaves when clients are slow, downstreams are slow, payloads are large, deploys are mixed-version, and connections are drained.

## Performance implications

Clean boundaries make performance work local. If protocol parsing, business logic, and outbound writes are tangled, every optimization risks changing behavior.

## Common bugs

- drawing services but not queues
- defining protocols without versioning
- allowing unbounded fanout
- hiding blocking work in event handlers
- treating retries as reliability without budgets

## Summary

Networking architecture is the design of byte contracts under pressure. Make the contracts explicit and the runtime becomes easier to operate.
