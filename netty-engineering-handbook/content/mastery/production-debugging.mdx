<StoryBlock title="Debug The Boundary That Failed">
Production networking incidents become manageable when you identify the failed boundary: accept, connect, TLS, decode, handler, downstream, write, or close.
</StoryBlock>

## Intuition

A network outage is rarely one signal. It is a stack of symptoms across clients, load balancers, kernels, runtimes, and application handlers.

Good debugging narrows the layer before changing the system.

<Misconception
  soundsLike="Production debugging means trying the likely fix quickly."
  actually="Production debugging means narrowing the failed boundary with evidence, changing the smallest useful control, and verifying the original symptom."
  whyItMatters="Fast guesses often destroy evidence, widen blast radius, or make a queueing incident look temporarily better while the cause remains."
/>

## Real-world problem

Clients report timeouts. Server CPU is low. The load balancer shows many open connections. Netty write buffers are high. The actual issue is slow clients plus missing backpressure.

## Mental model

Follow the lifecycle:

- can clients connect?
- can TLS or protocol handshake complete?
- can frames decode?
- can handlers finish quickly?
- can writes drain?
- do closes look intentional?

Debugging is a narrowing exercise. A good incident note should say: **we first saw symptom X, ruled out layers A and B, confirmed layer C with signal Y, changed Z, and verified with metric W.**

## Runtime behavior

Netty gives useful signals if you expose them: EventLoop delay, pending tasks, channel counts, allocator usage, write buffer bytes, decode failures, and exception types.

<RuntimeTrace title="Layered evidence ladder">
Client | Error type, timeout phase, retry behavior, connection reuse.
Load balancer | Open connections, resets, 4xx/5xx shape, backend health.
Kernel | SYN backlog, accept queue, retransmits, socket states.
Runtime | EventLoop delay, pending tasks, direct memory, write buffer growth.
Protocol | Decode failures, invalid frames, handshake failures, close reasons.
Application | Handler latency, downstream calls, queue depth, side-effect errors.
</RuntimeTrace>

## Code example

```java
import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.ChannelInboundHandlerAdapter;
import java.net.SocketAddress;

public final class ConnectionTelemetryHandler extends ChannelInboundHandlerAdapter {
    private final Metrics metrics;

    public ConnectionTelemetryHandler(Metrics metrics) {
        this.metrics = metrics;
    }

    @Override
    public void channelActive(ChannelHandlerContext ctx) {
        metrics.counter("netty.channel.active").increment();
        ctx.fireChannelActive();
    }

    @Override
    public void channelInactive(ChannelHandlerContext ctx) {
        metrics.counter("netty.channel.inactive").increment();
        ctx.fireChannelInactive();
    }

    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        SocketAddress remote = ctx.channel().remoteAddress();

        metrics.counter("netty.pipeline.exception",
            "handler", getClass().getSimpleName(),
            "type", cause.getClass().getSimpleName(),
            "remoteKnown", String.valueOf(remote != null)
        ).increment();

        ctx.close();
    }
}

interface Metrics {
    Counter counter(String name, String... tags);
}

interface Counter {
    void increment();
}
```

The handler name and exception type help locate the failing boundary.

<Warning title="Do not debug by leaking data">
During incidents, log protocol phase, handler name, channel id, remote presence, error class, and safe correlation ids. Avoid raw payloads, tokens, cookies, credentials, or full personal data unless you have a controlled redaction path.
</Warning>

<RecallPrompt question="what makes an incident note useful after the outage?">
It names the first symptom, the layers ruled out, the signal that confirmed the failed boundary, the change made, and the metric that proved recovery.
</RecallPrompt>

## Production implications

Incident dashboards should follow the connection lifecycle. Logs should preserve remote address, channel id, protocol phase, and failure reason without exposing sensitive payloads.

## Performance implications

Debug instrumentation must be cheap by default. Expensive wire logging belongs behind sampling, feature flags, or targeted reproduction.

## Common bugs

- changing timeouts before identifying the stuck queue
- logging raw payloads during incidents
- ignoring client-side and load-balancer evidence
- treating all disconnects as errors
- forgetting to verify the fix under load

## Summary

Production debugging is a layering exercise. Find the boundary, gather evidence, change the smallest thing, and verify runtime behavior.
