<StoryBlock title="Understanding Needs Evidence">
You do not really understand a system when the happy path works once. You understand it when you can break it, observe the failure, explain the runtime story, and then make a narrow fix.
</StoryBlock>

## Intuition

A good project is not a big app. It is a small system with one sharp lesson: bytes can split, consumers can lag, indexes can miss, cache keys can go stale, containers can exit, and WebSocket connections can live on the wrong server.

The project should make the hidden part visible.

<Misconception
  soundsLike="A good learning project is a large app with many features."
  actually="A good systems project is a small, observable failure lab that proves one runtime rule under breakage."
  whyItMatters="Feature-heavy demos can hide whether you understand queues, bytes, offsets, indexes, cache pressure, containers, or connection ownership."
/>

## Real-world problem

Most tutorial projects stop too early. They prove that an API can be called, but not that you understand what happens under pressure.

That is why a portfolio of small failure-driven projects teaches more than one large demo. Each project should answer four questions:

- What runtime rule am I proving?
- What failure will expose the rule?
- What evidence will I capture?
- What design change will I defend after seeing the evidence?

## Mental model

Pick the project by the failure you need to understand.

If you are studying Netty, build a framed protocol and break message boundaries. If you are studying Kafka, build replay and lag drills. If you are studying MongoDB, prove an index or transaction boundary. If you are studying Redis, create a hot-key expiry. If you are studying Docker, debug a container as a process. If you are studying WebSocket, scale past one server.

<Warning title="A project without failure is a demo">
If the project does not include malformed input, slow consumers, restart, overload, stale data, bad configuration, or observability, it will make the happy path familiar and leave the production lesson missing.
</Warning>

## Runtime behavior

Every project should force one runtime boundary into the open.

| Track | Runtime boundary to expose | Failure to inject | Evidence to keep |
| --- | --- | --- | --- |
| Netty | TCP bytes becoming application messages | sticky packets, split frames, slow handler | decoder logs, EventLoop thread names, latency |
| Kafka | records moving through partitions and offsets | slow consumer, rebalance, replay | lag per partition, assignments, offset reset output |
| MongoDB | query shape meeting index shape | unsupported sort, wrong index order, half-failed workflow | explain plan, documents examined, invariant failure |
| Redis | cache and memory pressure | hot-key expiry, stampede, unacked stream message | hit rate, source-call count, pending entries |
| Docker | image artifact becoming one process | missing env var, wrong command, bad dependency | exit code, logs, image history, inspect output |
| WebSocket | persistent connection ownership | malformed message, server kill, two-instance broadcast | 101 handshake, close code, delivery matrix |

<RuntimeTrace title="Evidence-first project loop">
Choose rule | Pick one runtime rule the project must prove.
Build path | Implement the smallest happy path that exposes the rule.
Break path | Inject the failure that makes the hidden boundary visible.
Capture evidence | Save metrics, logs, commands, traces, or screenshots.
Explain fix | Make one design change and describe why that evidence justifies it.
Repeat | Add the next failure only after the first lesson is observable.
</RuntimeTrace>

## Code example

The repository includes a runnable Spring Boot lab at `examples/systems-lab`. It is intentionally small: one service, four production-style integrations, and commands that let you prove the behavior instead of only reading about it.

```bash
cd examples/systems-lab
mvn -q -DskipTests package
docker compose up --build
```

After it starts, use the endpoints in the lab README to send a Kafka order event, read a Redis-backed product, seed MongoDB data, inspect an explain plan, and open the WebSocket/STOMP notification path. Treat it as the shared practice bench for the Kafka, Redis, MongoDB, Docker, and WebSocket tracks.

```text
Project brief: evidence-first Redis cache

Build:
  1. read product by key from Redis
  2. load from source on miss
  3. write with TTL and small jitter
  4. expose hit, miss, source-call count, and latency

Break:
  1. expire one hot key under concurrent traffic
  2. slow the source call
  3. remove the stampede guard

Observe:
  1. source calls before and after the guard
  2. p95 latency during refill
  3. stale-read behavior when the source is slow

Explain:
  A cache miss can amplify traffic. The fix is not only "add Redis";
  the fix is a controlled refill policy with bounded waiting or stale data.
```

Use this same shape for every track. The language changes, but the engineering loop stays the same.

<RecallPrompt question="why should every practice project include a failure case?">
The failure case proves the runtime model. Without it, the project only shows that the happy path can run once.
</RecallPrompt>

## Production implications

Turn each project into a small runbook:

- symptoms you expect to see
- metrics and logs you need first
- commands or dashboards that prove the state
- one safe mitigation
- one deeper fix
- one anti-pattern you will avoid next time

This matters because production incidents punish vague understanding. A runbook forces the project to become operational knowledge.

## Performance implications

Benchmark only after correctness and bounds exist. Performance measurements without failure cases usually reward the wrong thing.

For Netty and WebSocket, watch latency distribution, write queues, EventLoop delay, and memory. For Kafka, watch producer errors, broker latency, lag, and rebalance frequency. For MongoDB, watch documents examined, index usage, lock or write pressure, and slow queries. For Redis, watch command latency, hit rate, memory, evictions, and source-call amplification. For Docker, watch startup time, image size, health checks, CPU, memory, and network assumptions.

## Common bugs

- building only happy paths
- skipping malformed input tests
- treating "it ran once" as proof
- collecting logs without deciding what question they answer
- adding scale before understanding the single-node failure
- benchmarking before adding bounds and observability
- writing a project README that explains features but not failures

## Summary

Real projects should make runtime behavior visible. Build small systems that force you to reason about bytes, queues, partitions, indexes, caches, processes, connections, and failure.
