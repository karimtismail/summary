<StoryBlock title="Understanding Needs A Runtime">
You do not really understand Netty by reading APIs. You understand it by building something that fails until the mental model is correct.
</StoryBlock>

## Intuition

Practice should follow the same learning flow as the handbook: blocking IO, selectors, EventLoops, pipelines, framing, backpressure, and production debugging.

## Real-world problem

Tutorial projects often stop at echo servers. Echo servers teach connection setup, but they do not teach partial frames, slow clients, memory ownership, protocol evolution, or operational pressure.

## Mental model

Choose projects that force a specific runtime lesson. Keep the feature small and the behavior sharp.

Do not ask "what can I build?" first. Ask "what failure do I need to understand?" Then build the smallest system that makes that failure observable.

<ArchitectureDiagram title="Project ladder">

  proves: writability, backpressure, bounded queues

  proves: metrics, logs, failure injection, runbook thinking
</ArchitectureDiagram>

## Runtime behavior

Each project should include fault injection: partial writes, slow consumers, malformed frames, reconnect storms, large payloads, and downstream delay. The failure is the teacher.

<RuntimeTrace title="A project review loop">
Build | Implement the smallest behavior that exposes one runtime rule.
Break | Add a failure case that should stress the rule.
Observe | Capture metrics, logs, thread names, memory, and protocol phase.
Explain | Write the failure story before fixing it.
Harden | Add bounds, tests, and a runbook note.
</RuntimeTrace>

## Code example

```text
Project: backpressured room broadcast

Requirement:
  When one client stops reading, other clients must keep receiving messages.
  Memory must remain bounded.
  The slow client must be paused, downgraded, or closed by policy.

Failure injection:
  - one client reads one byte per second
  - one client sends malformed length-prefixed frames
  - one client reconnects every 100 ms
  - one producer sends a burst larger than the per-channel queue limit

Evidence:
  - channel writability changes
  - per-client queued messages
  - EventLoop delay
  - close reason
  - p95 and p99 delivery latency for healthy clients
```

This requirement teaches more than another happy-path endpoint.

<Warning title="A project without failure is a demo">
If the project does not include a slow client, malformed input, restart, overload, or observability requirement, it will make the happy path familiar but leave the production lesson missing.
</Warning>

## Production implications

Turn projects into runbooks. For each one, record metrics, failure modes, expected logs, and a debugging path.

## Performance implications

Benchmark only after correctness is stable. Then measure allocation rate, EventLoop delay, write buffer growth, and p99 latency under controlled pressure.

## Common bugs

- building only happy paths
- skipping malformed input tests
- ignoring slow readers
- benchmarking before adding bounds
- treating generated load as realistic traffic

## Summary

Real projects should make the invisible runtime visible. Build small systems that force you to reason about bytes, queues, threads, and failure.
