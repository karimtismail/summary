<StoryBlock title="A Topic Is Not One Big File">
Kafka scales a topic by splitting it into partitions. Each partition is an ordered log, and brokers distribute those logs across the cluster.
</StoryBlock>

## Intuition

A broker is a Kafka server. A topic is a named stream. A partition is the actual ordered log inside that stream.

Kafka gives ordering inside a partition, not across all partitions in a topic.

<Misconception
  soundsLike="A topic is one ordered stream, and adding partitions only makes it faster."
  actually="A topic is split into partition logs. Ordering is per partition, and each partition adds broker metadata, file, replication, recovery, and rebalance work."
  whyItMatters="Over-partitioning can hurt operations, while under-partitioning caps consumer parallelism."
/>

## Real-world problem

An event stream receives 100,000 order updates per second. One ordered log on one machine becomes a bottleneck. Splitting the topic into partitions lets many brokers store and serve the stream in parallel.

## Mental model

Partitions are the central unit:

- producers choose a partition
- brokers store partitions
- consumers read partitions
- ordering exists within a partition
- parallelism increases with partition count

## Runtime behavior

When a producer sends a record with a key, Kafka's partitioner maps that key to a partition. Records with the same key usually land on the same partition, preserving per-key order.

When a consumer group reads a topic, Kafka assigns partitions to group members.

<RuntimeTrace title="Partition as scaling unit">
Produce | The key and partitioner choose one partition for the record.
Append | The leader broker stores the record in that partition log.
Replicate | Followers copy the partition from the leader.
Assign | The consumer group gives the partition to one member at a time.
Scale | More partitions allow more active consumers, up to operational limits.
Recover | Failure or rebalance work happens per partition, not per abstract topic.
</RuntimeTrace>

## Code example

```java
ProducerRecord<String, String> record =
    new ProducerRecord<>("orders.events", customerId, eventJson);

// Same key usually maps to the same partition.
producer.send(record);
```

The key is not just metadata. It is often your ordering and scaling decision.

<RecallPrompt question="why can't a consumer group use more active consumers than partitions?">
Within one consumer group, each partition is assigned to only one active member at a time, so extra members wait idle until more partitions or assignment changes exist.
</RecallPrompt>

## Production implications

Choose partition keys by access pattern and ordering needs. A low-cardinality key can create hot partitions. A random key can destroy useful ordering.

## Performance implications

More partitions can increase parallelism, but they also increase metadata, open files, leader election work, and recovery cost. Partition count is capacity design, not a free slider.

## Common bugs

- expecting total topic ordering with multiple partitions
- using a low-cardinality key such as region for high-volume traffic
- creating too many tiny topics
- changing partition count without understanding key remapping
- assuming consumers scale beyond partition count

## Summary

Brokers host partitions, and partitions define Kafka's real ordering and scaling behavior. Design the partition key as carefully as an API.
