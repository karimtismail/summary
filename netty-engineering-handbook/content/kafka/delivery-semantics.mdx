<StoryBlock title="Every Guarantee Has A Failure Story">
At-most-once, at-least-once, and exactly-once are not slogans. They are answers to what happens around crashes and retries.
</StoryBlock>

## Intuition

Delivery semantics describe duplicate and loss behavior. Kafka gives primitives, but the end-to-end guarantee depends on producers, consumers, offset commits, side effects, and downstream systems.

<Misconception
  soundsLike="Kafka's delivery setting decides the whole application's correctness guarantee."
  actually="The guarantee is end to end: producer writes, topic durability, consumer offset timing, side effects, retries, and downstream idempotency all participate."
  whyItMatters="A workflow can use strong Kafka settings and still charge, email, or write to a database twice."
/>

## Real-world problem

A consumer charges a credit card and crashes before committing the offset. On restart, it reads the same event again. Kafka worked correctly. The application was not idempotent.

## Mental model

Semantics by failure mode:

- at-most-once: records may be lost, not duplicated
- at-least-once: records may duplicate, not silently disappear after processing starts
- exactly-once: duplicate effects are prevented inside a defined transactional boundary

The boundary is the whole point. Kafka can protect Kafka records and committed Kafka offsets. It cannot magically undo a charged card, an email that was already sent, or a database write that did not participate in the same transaction.

## Runtime behavior

Kafka can provide idempotent producers and transactions for Kafka-to-Kafka workflows. External systems still need idempotency keys, transactional writes, or reconciliation.

<RuntimeTrace title="End-to-end guarantee checklist">
Produce | Producer retries, acknowledgements, keys, and idempotence decide what can appear in the log.
Store | Topic replication and `acks` decide when a write is considered durable enough.
Read | Consumer group assignment and polling decide which instance sees a partition.
Process | Business code decides whether duplicate attempts are harmless.
Commit | Offset commit position decides where replay resumes after crash.
Side effect | External systems decide whether the same event id can apply twice.
</RuntimeTrace>

## Code example

```java
import org.apache.kafka.clients.consumer.ConsumerRecord;
import java.util.List;

public final class IdempotentOrderConsumer {
    private final ProcessedEventRepository processedEvents;
    private final PaymentGateway payments;

    public IdempotentOrderConsumer(ProcessedEventRepository processedEvents, PaymentGateway payments) {
        this.processedEvents = processedEvents;
        this.payments = payments;
    }

    public void processBatch(List<ConsumerRecord<String, OrderPlaced>> records) {
        for (ConsumerRecord<String, OrderPlaced> record : records) {
            OrderPlaced event = record.value();

            if (!processedEvents.tryMarkProcessing(event.eventId())) {
                continue;
            }

            payments.chargeOnce(event.eventId(), event.accountId(), event.amountCents());
            processedEvents.markDone(event.eventId());
        }
    }
}

record OrderPlaced(String eventId, String accountId, long amountCents) {}

interface ProcessedEventRepository {
    boolean tryMarkProcessing(String eventId);
    void markDone(String eventId);
}

interface PaymentGateway {
    void chargeOnce(String idempotencyKey, String accountId, long amountCents);
}
```

This application-level guard often matters more than the broker setting.

<Warning title="Commit position is not business correctness">
Committing an offset says "this consumer group can resume after this record." It does not prove the external side effect is correct. For external systems, use idempotency keys, unique constraints, outbox/inbox tables, or reconciliation.
</Warning>

<RecallPrompt question="where does exactly-once stop helping?">
Kafka transactions can protect Kafka records and Kafka offsets inside their boundary. External APIs, emails, payments, and unrelated databases still need idempotency or reconciliation.
</RecallPrompt>

## Production implications

Document each consumer's semantics. Payment, email, analytics, and cache invalidation do not need the same behavior. Make duplicate handling visible in logs and metrics.

## Performance implications

Stronger semantics usually add coordination, storage, or transactional cost. Use them where the business effect needs them, not everywhere by default.

## Common bugs

- claiming exactly-once for workflows that call external APIs
- committing offsets before processing
- missing idempotency keys
- retrying poison records forever
- using one error policy for all event types

## Summary

Delivery semantics are end-to-end properties. Kafka supplies tools, but the application decides whether duplicates become harmless or dangerous.
