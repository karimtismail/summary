<StoryBlock title="A Producer Is A Write Pipeline">
The producer is not just a client library. It batches, partitions, retries, compresses, and waits for acknowledgements.
</StoryBlock>

## Intuition

Producing to Kafka is an async append workflow. The application creates a record, the producer assigns a partition, batches it with nearby records, sends it to the leader broker, and receives an acknowledgement.

<Misconception
  soundsLike="Calling `send` means the event is already safely in Kafka."
  actually="`send` usually enqueues work in the producer. The durable outcome arrives later through metadata, callback, future completion, or an error path."
  whyItMatters="Ignoring async failures or shutting down too early can lose events even though the application called the producer API."
/>

## Real-world problem

A checkout service sends one Kafka request per order event with no batching and weak acknowledgements. Latency looks simple in development, but production throughput is poor and failures lose records.

## Mental model

Producer behavior is shaped by four questions:

- what partition should receive this record?
- how long can the producer wait to batch?
- how many replicas must acknowledge?
- what happens when the send fails?

## Runtime behavior

The Kafka producer has background IO threads. `send` usually returns quickly with a future. Actual network writes happen asynchronously. Retries can reorder records if idempotence is disabled and multiple in-flight requests are allowed for the same connection.

In Kafka 4.2, `acks=all`, `enable.idempotence=true`, and `linger.ms=5` are defaults when no conflicting producer settings disable them. The example keeps them explicit because correctness settings should be visible in code review.

<RuntimeTrace title="Producer append lifecycle">
Create | Application builds a record with topic, key, value, and headers.
Partition | The producer chooses the partition that owns ordering for this key.
Batch | Records wait briefly or until a batch is ready.
Send | The IO thread sends the batch to the partition leader.
Ack | Broker response confirms success or exposes retryable/non-retryable failure.
Callback | Application code records success, handles error, or trips its own alert.
</RuntimeTrace>

## Code example

The runnable lab in `examples/systems-lab` includes a Spring Boot Kafka producer endpoint:

```bash
curl -X POST http://localhost:8080/kafka/orders/order-101
```

Use it to connect this chapter to a real send path: controller -> `KafkaTemplate` -> topic partition -> listener evidence.

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("acks", "all");
props.put("enable.idempotence", "true");
props.put("linger.ms", "5");
props.put("compression.type", "zstd");

KafkaProducer<String, String> producer = new KafkaProducer<>(
    props,
    new StringSerializer(),
    new StringSerializer()
);
```

These settings express a tradeoff between durability, throughput, latency, and duplicate behavior.

<RecallPrompt question="why should producer callbacks be handled even when `send` returned?">
The send call can succeed locally while the later broker write fails, times out, or is retried. The callback or future is where the durable outcome becomes visible.
</RecallPrompt>

## Production implications

Producers need error handling, metrics, delivery callbacks, bounded shutdown, and clear behavior when Kafka is unavailable. Silent async failures are one of the most common production mistakes.

## Performance implications

Batching and compression drive throughput. `linger.ms`, `batch.size`, and compression type should be tested with realistic payloads and latency requirements.

## Common bugs

- ignoring the callback from `send`
- using `acks=0` for important events
- assuming current idempotence defaults still apply after changing retries, acks, or in-flight limits
- calling `flush` after every record
- choosing keys that create hot partitions
- shutting down before in-flight records finish

## Summary

Kafka producers are durable async write pipelines. Treat their configuration as part of your correctness and performance design.
