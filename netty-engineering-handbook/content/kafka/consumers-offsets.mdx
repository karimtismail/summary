<StoryBlock title="A Consumer Is A Cursor">
Kafka does not remove a record when a consumer sees it. The consumer advances its own cursor through the log.

The cursor analogy has limits. The cursor is per consumer group and partition, retention can remove old records, and committing the cursor says nothing about whether your database write, API call, or email already succeeded.
</StoryBlock>

## Intuition

Offsets are Kafka's progress mechanism. A consumer reads records from a partition and commits the offset it has processed. Replay is possible because the log remains available until retention removes it.

<Misconception
  soundsLike="Committing an offset means the business work definitely finished."
  actually="Committing an offset only records consumer progress. Your side effect may have failed, duplicated, or still be in flight."
  whyItMatters="Offset timing decides whether a crash turns into loss, duplicate work, or a recoverable replay."
/>

## Real-world problem

A payment consumer reads a record, commits the offset, then fails before updating the database. On restart, Kafka thinks progress was made and the payment event is skipped.

## Mental model

Offset commit timing defines failure behavior:

- commit before processing: possible loss
- commit after processing: possible duplicates
- commit transactionally with output: stronger workflows

Commit before side effect | Fast progress, but a crash can skip work permanently.
Side effect before commit | Replay can repeat work, so handlers need idempotency.
Batch partially processed | A commit for the whole batch can hide unfinished records.
Rebalance during work | Another consumer may receive the partition while old work is still in flight.
</RuntimeTrace>

## Runtime behavior

Consumers in the same group divide partitions among themselves. If a consumer joins, leaves, or stops heartbeating, Kafka rebalances partition assignments.

In Kafka 4.2, the classic group protocol remains the default, and Kafka also supports the newer `consumer` group protocol. Heartbeat and session-timeout ownership differs between those modes, so copy consumer tuning from the protocol you actually run.

Rebalances are normal, but they interrupt work if offset and processing boundaries are sloppy. If you commit offsets yourself, disable background auto-commit first.

## Code example

```java
props.put("enable.auto.commit", "false");

while (running) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(500));

    for (ConsumerRecord<String, String> record : records) {
        handle(record.key(), record.value());
    }

    consumer.commitSync();
}
```

This commits after processing the batch, which usually means duplicates are possible after a crash. Without `enable.auto.commit=false`, a background commit may record progress before your processing boundary is complete.

<RecallPrompt question="why are duplicates often safer than loss?">
Duplicates can be made harmless with idempotency keys, processed-event tables, or reconciliation. A lost event may never be seen again because the committed offset tells Kafka to resume after it.
</RecallPrompt>

## Production implications

Handlers must be idempotent unless the full workflow provides stronger guarantees. Track consumer lag, rebalance frequency, poll duration, processing failures, and dead-letter volume.

## Performance implications

Throughput depends on fetch sizing, batch processing, partition count, handler speed, and downstream capacity. A slow downstream system becomes Kafka lag.

## Common bugs

- committing offsets before side effects complete
- mixing manual commits with background auto-commit
- treating consumer lag as always bad without understanding replay
- blocking too long between polls
- assuming adding consumers helps beyond partition count
- not handling rebalance callbacks for in-flight work

## Summary

Consumers are cursors through durable logs. Offset strategy is the bridge between replay power and processing correctness.
