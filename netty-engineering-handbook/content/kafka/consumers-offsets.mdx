title: Consumers and Offsets
description: Consumers read partitions and use offsets to record progress, replay, and recovery.

<StoryBlock title="A Consumer Is A Cursor">
Kafka does not remove a record when a consumer sees it. The consumer advances its own cursor through the log.
</StoryBlock>

## Intuition

Offsets are Kafka's progress mechanism. A consumer reads records from a partition and commits the offset it has processed. Replay is possible because the log remains available until retention removes it.

## Real-world problem

A payment consumer reads a record, commits the offset, then fails before updating the database. On restart, Kafka thinks progress was made and the payment event is skipped.

## Mental model

Offset commit timing defines failure behavior:

- commit before processing: possible loss
- commit after processing: possible duplicates
- commit transactionally with output: stronger workflows

## Runtime behavior

Consumers in the same group divide partitions among themselves. If a consumer joins, leaves, or stops heartbeating, Kafka rebalances partition assignments.

Rebalances are normal, but they interrupt work if offset and processing boundaries are sloppy.

## Code example

```java
while (running) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(500));

    for (ConsumerRecord<String, String> record : records) {
        handle(record.key(), record.value());
    }

    consumer.commitSync();
}
```

This commits after processing the batch, which usually means duplicates are possible after a crash.

## Production implications

Handlers must be idempotent unless the full workflow provides stronger guarantees. Track consumer lag, rebalance frequency, poll duration, processing failures, and dead-letter volume.

## Performance implications

Throughput depends on fetch sizing, batch processing, partition count, handler speed, and downstream capacity. A slow downstream system becomes Kafka lag.

## Common bugs

- committing offsets before side effects complete
- treating consumer lag as always bad without understanding replay
- blocking too long between polls
- assuming adding consumers helps beyond partition count
- not handling rebalance callbacks for in-flight work

## Summary

Consumers are cursors through durable logs. Offset strategy is the bridge between replay power and processing correctness.
