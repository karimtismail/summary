<StoryBlock title="A Stream Processor Is A Database In Motion">
Stateful stream processing is not just consuming messages quickly. It maintains evolving state from an event log.
</StoryBlock>

## Intuition

Stream processing reads events continuously, updates state, and emits new events. It is useful for aggregations, enrichment, joins, fraud signals, materialized views, and real-time workflows.

<Misconception
  soundsLike="A stream processor is just a faster consumer loop."
  actually="A stateful stream processor owns topology, local state, changelog recovery, time semantics, and output contracts."
  whyItMatters="Ignoring state and recovery makes a topology look fast until a rebalance, restart, or late event exposes the hidden cost."
/>

## Real-world problem

A dashboard needs order totals per merchant every minute. Querying the transactional database for every refresh is expensive and delayed. A stream processor can maintain the running aggregate as events arrive.

## Mental model

Think in four parts:

- source topics
- processing topology
- local state stores
- output topics and changelogs

State is recoverable because Kafka keeps the input and changelog history.

The operational question is: **how long would it take to rebuild this state after a node dies?** If that answer is surprising, the topology is not production-ready yet.

## Runtime behavior

Kafka Streams assigns input partitions to tasks. Tasks process records, update local state, write changelogs, and produce output. On failure, another instance restores state from changelog topics.

<RuntimeTrace title="Stateful task lifecycle">
Assign | Kafka Streams assigns input partitions to stream tasks.
Process | Each task reads records in partition order.
Update state | Local stores keep aggregates, join tables, or windows.
Changelog | Store updates are written to compacted changelog topics.
Commit | Offsets and state progress are checkpointed.
Restore | After rebalance or crash, another task rebuilds state before processing resumes.
</RuntimeTrace>

## Code example

```java
import org.apache.kafka.common.serialization.Serdes;
import org.apache.kafka.streams.StreamsBuilder;
import org.apache.kafka.streams.kstream.Consumed;
import org.apache.kafka.streams.kstream.Materialized;
import org.apache.kafka.streams.kstream.Produced;
import org.apache.kafka.streams.kstream.TimeWindows;
import org.apache.kafka.streams.kstream.Windowed;
import java.time.Duration;

public final class MerchantOrderTopology {
    public static void build(
        StreamsBuilder builder,
        OrderSerde orderSerde,
        org.apache.kafka.common.serialization.Serde<Windowed<String>> windowedMerchantSerde
    ) {
        builder.stream("orders", Consumed.with(Serdes.String(), orderSerde))
            .groupBy((key, order) -> order.merchantId())
            .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(1)))
            .count(Materialized.as("merchant-order-counts"))
            .toStream()
            .to("merchant.order-counts", Produced.with(windowedMerchantSerde, Serdes.Long()));
    }
}

interface OrderSerde extends org.apache.kafka.common.serialization.Serde<Order> {}
record Order(String merchantId) {}
```

This is a continuous computation, not a batch query.

<Warning title="Window choice is business semantics">
Tumbling, hopping, and session windows answer different product questions. Also decide how late events are handled; processing time and event time are not the same thing.
</Warning>

<Highlight title="Recovery is part of performance">
State store size, changelog compaction, local disk speed, standby replicas, and checkpoint freshness decide how long a rebalance or restart hurts the application.
</Highlight>

<RecallPrompt question="why is restore time part of stream-processing performance?">
A stateful task cannot resume useful work until its local state is available again, so changelog size, disk speed, and standby strategy affect incident duration and rebalance pain.
</RecallPrompt>

## Production implications

Stateful processors need deployment discipline: standby replicas, changelog retention, restore time planning, poison-record handling, and versioned state migrations.

## Performance implications

Performance depends on partition count, state size, RocksDB behavior, cache settings, commit interval, serialization, and output batching. Restore time is part of performance.

## Common bugs

- treating event time and processing time as the same
- ignoring late events
- making state stores too large to restore quickly
- changing state format without migration
- joining streams without understanding key alignment

## Summary

Stream processing turns Kafka logs into live state. The advanced skill is managing time, state, and recovery deliberately.
