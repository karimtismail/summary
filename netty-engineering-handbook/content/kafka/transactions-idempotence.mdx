title: Transactions and Idempotence
description: Idempotent producers and transactions reduce duplicates and tie Kafka reads to Kafka writes.

<StoryBlock title="Duplicates Are A Design Input">
Distributed systems retry. Retries create duplicate attempts. Kafka's advanced producer features help when duplicates must not become duplicate log entries or output effects.
</StoryBlock>

## Intuition

Idempotence prevents duplicate records caused by producer retries within a producer session. Transactions let a consumer read input records, produce output records, and commit offsets as one atomic Kafka transaction.

<Misconception
  soundsLike="Turning on transactions makes the whole workflow exactly once."
  actually="Kafka transactions make Kafka input offsets and Kafka output records atomic inside the transaction boundary. External side effects remain outside unless you design for them."
  whyItMatters="A handler can avoid duplicate Kafka output but still duplicate a database write or external API call."
/>

## Real-world problem

A stream processor reads orders and writes invoices. It crashes after writing invoices but before committing input offsets. Without transactions, replay can write duplicate invoices to the output topic.

## Mental model

Idempotence protects producer retries. Transactions protect Kafka-to-Kafka processing boundaries.

They do not automatically make external side effects exactly once.

## Runtime behavior

A transactional producer has a `transactional.id`. Kafka uses producer epochs and transaction markers to fence old producer instances and expose committed records to consumers using `read_committed`.

<RuntimeTrace title="Transactional consume-process-produce">
Initialize | The producer registers transactional identity and gets fenced if an older instance is unsafe.
Begin | A transaction starts before output records are produced.
Produce | Output records are written but not visible to `read_committed` consumers yet.
Offsets | The input offsets are sent to the same transaction.
Commit | Output records and offset progress become visible together.
Abort | Output records stay hidden and offsets do not advance.
</RuntimeTrace>

## Code example

```java
import org.apache.kafka.clients.consumer.ConsumerGroupMetadata;
import org.apache.kafka.clients.consumer.ConsumerRecord;
import org.apache.kafka.clients.consumer.OffsetAndMetadata;
import org.apache.kafka.clients.producer.KafkaProducer;
import org.apache.kafka.clients.producer.ProducerRecord;
import org.apache.kafka.common.TopicPartition;
import java.util.List;
import java.util.Map;

public final class TransactionalInvoiceProcessor {
    private final KafkaProducer<String, InvoiceCreated> producer;

    public TransactionalInvoiceProcessor(KafkaProducer<String, InvoiceCreated> producer) {
        this.producer = producer;
        this.producer.initTransactions();
    }

    public void process(
        List<ConsumerRecord<String, OrderPaid>> records,
        Map<TopicPartition, OffsetAndMetadata> offsets,
        ConsumerGroupMetadata groupMetadata
    ) {
        producer.beginTransaction();
        try {
            for (ConsumerRecord<String, OrderPaid> record : records) {
                OrderPaid order = record.value();
                producer.send(new ProducerRecord<>(
                    "invoice-created",
                    order.orderId(),
                    new InvoiceCreated(order.orderId(), order.amountCents())
                ));
            }

            producer.sendOffsetsToTransaction(offsets, groupMetadata);
            producer.commitTransaction();
        } catch (RuntimeException error) {
            producer.abortTransaction();
            throw error;
        }
    }
}

record OrderPaid(String orderId, long amountCents) {}
record InvoiceCreated(String orderId, long amountCents) {}
```

The output records and offset progress commit together.

Consumers that depend on this output should set `isolation.level=read_committed`; otherwise they may observe records from transactions that later abort.

<Warning title="External writes are outside this transaction">
This pattern protects Kafka input offsets and Kafka output records. If the same handler also writes to PostgreSQL, MongoDB, Redis, or an external API, that side effect needs its own idempotency or an outbox-style design.
</Warning>

<RecallPrompt question="why should downstream consumers use `read_committed`?">
Without `read_committed`, consumers may observe records from transactions that later abort, which breaks the boundary the transactional producer was trying to provide.
</RecallPrompt>

## Production implications

Transactional IDs must be stable per processing instance and fenced correctly during deploys. Consumers that require committed output should use `isolation.level=read_committed`.

## Performance implications

Transactions add coordination and can reduce throughput if batches are too small. The transaction boundary should contain meaningful work without becoming too large to recover cleanly.

## Common bugs

- using random `transactional.id` values on restart
- expecting transactions to cover database writes automatically
- forgetting `read_committed` on downstream consumers
- making one enormous transaction
- not handling producer fencing errors

## Summary

Idempotence and transactions are powerful Kafka guarantees within clear boundaries. Use them for duplicate-resistant Kafka workflows and pair them with application idempotency elsewhere.
