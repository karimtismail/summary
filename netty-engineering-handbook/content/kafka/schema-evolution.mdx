<StoryBlock title="Events Outlive Deploys">
A Kafka event may be read by services that deploy tomorrow, next month, or during a replay next year. Schema evolution is how that history remains useful.
</StoryBlock>

## Intuition

Kafka stores bytes. The meaning of those bytes lives in a schema contract. If producers and consumers disagree about that contract, replay becomes dangerous.

<Misconception
  soundsLike="Schema changes are safe if the current producer and current consumer deploy together."
  actually="Kafka history can be read by old consumers, new consumers, replay jobs, and delayed systems, so compatibility must include old records and mixed deployments."
  whyItMatters="A change that passes today's happy path can break replay, dead-letter recovery, analytics, or a slow consumer group later."
/>

## Real-world problem

A producer renames `customerId` to `accountId`. New consumers work. Old consumers fail. Replaying old records into new code also fails because historical events still contain the old field.

## Mental model

Design for compatibility:

- add optional fields before requiring them
- keep old fields during migration
- avoid changing field meaning in place
- version events by contract, not just code release
- test old data with new consumers

Kafka decouples producers and consumers, so schema changes are multi-deploy migrations. A safe change usually means consumers become tolerant before producers become ambitious.

## Runtime behavior

Schema Registry systems can enforce compatibility for Avro, Protobuf, or JSON Schema. The broker stores bytes, but registry checks can prevent incompatible producers from writing.

Kafka itself does not include a schema registry, but production Kafka platforms commonly pair topics with a registry that stores historical schemas, applies compatibility rules, and lets serializers identify which schema wrote each record.

<RuntimeTrace title="Schema evolution rollout">
Propose | Add the new field or event version in the schema repository.
Check | CI verifies compatibility against historical schemas.
Tolerate | Consumers deploy code that can read old and new shapes.
Produce | Producers start writing the new shape while preserving old contracts.
Observe | Consumer errors, dead letters, and replay tests confirm safety.
Retire | Old fields disappear only after retention, replay, and consumer policy allow it.
</RuntimeTrace>

## Code example

```json
{
  "type": "record",
  "name": "OrderCreated",
  "fields": [
    { "name": "orderId", "type": "string" },
    { "name": "customerId", "type": "string" },
    { "name": "salesChannel", "type": ["null", "string"], "default": null }
  ]
}
```

Adding a nullable field with a default is a common compatible change.

```java
public record OrderCreatedV2(
    String orderId,
    String customerId,
    String accountId,
    String salesChannel
) {
    public String stableAccountId() {
        return accountId != null ? accountId : customerId;
    }
}
```

The compatibility adapter keeps old records readable while the organization migrates from `customerId` to `accountId`.

<Warning title="Do not redefine meaning in place">
Changing `customerId` from a customer identifier to an account identifier without a migration field is worse than a rename: the old consumers may keep running while silently making wrong decisions.
</Warning>

<RecallPrompt question="why is a rename usually a migration, not a one-line schema edit?">
Old records keep the old field, old consumers may still read the topic, and replay jobs need both meanings to stay explicit until retention and consumer policy allow removal.
</RecallPrompt>

## Production implications

Schema ownership should be explicit. Producers own meaning, but consumers need review paths for breaking changes. Compatibility checks belong in CI and deployment gates.

## Performance implications

Binary formats reduce payload size and parsing cost, but schema registry calls and serializer setup need careful caching. Do not trade clarity for micro-optimizations too early.

## Common bugs

- changing field meaning without changing the contract
- assuming new code only reads new events
- using unstructured JSON for critical business streams
- deleting fields before retention windows expire
- testing only producer serialization, not consumer replay

## Summary

Kafka events are durable contracts. Schema evolution keeps those contracts useful as systems change.
