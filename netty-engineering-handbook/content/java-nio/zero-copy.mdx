<StoryBlock title="The Best Copy Is The One You Never Schedule">
For large byte streams, performance often improves when the application stops touching bytes it does not need to understand.
</StoryBlock>

## Intuition

Zero copy means avoiding redundant copies between kernel space and user space. It does not mean no bytes move anywhere. It means the application does not copy bytes merely to pass them along.

## Real-world problem

A file server reads file bytes into a heap array and writes them to a socket. The application never inspects the payload. It pays allocation, copy, and cache costs for no product value.

## Mental model

If your application does not need to parse or transform a payload, keep it out of application memory when possible.

Zero copy is an architectural fit, not a decoration. It fits byte-moving services: static file serving, replication, proxying, and log shipping. It does not fit code that must inspect, compress, redact, encrypt in user space, or reshape every byte.

## Runtime behavior

Java exposes zero-copy file transfer through `FileChannel.transferTo` and related APIs. Netty exposes this idea with file region support when the transport and pipeline allow it.

TLS can change the story because bytes must be encrypted before leaving the process boundary.

<RuntimeTrace title="transferTo lifecycle">
Open | A file channel points at bytes already managed by the OS.
Attempt | `transferTo(position, count, target)` asks the runtime and OS to move bytes to the target channel.
Partial | The returned count may be smaller than requested because of file size, socket capacity, or non-blocking state.
Record | The application updates the file position it still needs to send.
Resume | The next writable event continues from the saved position.
</RuntimeTrace>

## Code example

```java
import java.io.IOException;
import java.nio.channels.FileChannel;
import java.nio.channels.WritableByteChannel;

public final class FileTransferState {
    private final FileChannel file;
    private long position;

    public FileTransferState(FileChannel file) {
        this.file = file;
    }

    public boolean transferTo(WritableByteChannel target) throws IOException {
        long size = file.size();
        while (position < size) {
            long sent = file.transferTo(position, size - position, target);
            if (sent == 0) {
                return false;
            }
            position += sent;
        }
        return true;
    }
}
```

The object keeps transfer progress. A non-blocking socket may accept zero bytes now, so the caller must keep this state and resume when the channel is writable again.

<Warning title="Zero copy does not remove backpressure">
The socket can still be full, the client can still be slow, and the transfer can still be partial. Treat file transfer as a paced outbound operation, not an instant side effect.
</Warning>

## Production implications

Zero copy is most useful for static files, replication streams, and payload forwarding. It is less useful when every byte must be parsed, encrypted in user space, compressed, or transformed.

## Performance implications

Benefits can include lower CPU, fewer allocations, less cache pollution, and higher throughput. Measure under realistic TLS, kernel, filesystem, and network conditions.

## Common bugs

- assuming zero copy works the same with TLS
- forgetting partial `transferTo` progress
- using it for small payloads where setup cost dominates
- ignoring OS-specific behavior
- losing backpressure because file transfer is treated as instantaneous

## Summary

Zero copy is a runtime shortcut around unnecessary application memory. Use it when the application is a byte mover, not a byte interpreter.
