<StoryBlock title="Slow Consumers Are Normal">
Backpressure is not an error path. It is the control system that keeps normal slowness from becoming an outage.
</StoryBlock>

## Intuition

Every network system has a point where it must say "not yet." Without that signal, memory grows, latency expands, retries multiply, and failure spreads.

<Misconception
  soundsLike="Backpressure is an error case you handle after the main path works."
  actually="Backpressure is the normal control path for mismatched speeds between producers, buffers, consumers, and downstream systems."
  whyItMatters="If overload has no planned signal, the real policy becomes unbounded memory growth, retry storms, or sudden connection loss."
/>

## Real-world problem

A server broadcasts updates to thousands of clients. Most keep up. A few mobile clients stall. If the server queues forever, slow clients consume memory that should serve healthy clients.

## Mental model

Backpressure connects producer speed to consumer capacity. It can pause reads, stop accepting work, drop optional messages, close abusive connections, or ask upstream systems to slow down.

## Visual explanation

<TracePrompt>
Trace where new work is created and where capacity is exhausted. Useful backpressure travels upstream to the producer of work, not only to the nearest buffer.
</TracePrompt>

<FlowDiagram title="Pressure path">
Consumer slows -> Write buffer grows -> High watermark reached -> Stop reading or producing -> Drain -> Resume
</FlowDiagram>

## Runtime behavior

Netty channels expose writability based on write buffer watermarks. Handlers can respond to `channelWritabilityChanged` and adjust production or reads.

## Code example

```java
import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.ChannelInboundHandlerAdapter;
import io.netty.channel.ChannelOption;
import io.netty.channel.socket.SocketChannel;
import io.netty.channel.WriteBufferWaterMark;

public final class BackpressureInitializer {
    public void configure(SocketChannel ch) {
        ch.config().setOption(
            ChannelOption.WRITE_BUFFER_WATER_MARK,
            new WriteBufferWaterMark(64 * 1024, 256 * 1024)
        );
        ch.pipeline().addLast("backpressure", new WritabilityBackpressureHandler());
    }
}

final class WritabilityBackpressureHandler extends ChannelInboundHandlerAdapter {
    @Override
    public void channelWritabilityChanged(ChannelHandlerContext ctx) {
        boolean writable = ctx.channel().isWritable();

        ctx.channel().config().setAutoRead(writable);
        ctx.fireChannelWritabilityChanged();
    }

    @Override
    public void channelInactive(ChannelHandlerContext ctx) {
        ctx.channel().config().setAutoRead(true);
        ctx.fireChannelInactive();
    }
}
```

This links outbound pressure to inbound demand: when queued outbound bytes cross the high watermark, the channel stops reading more input; when the queue drains below the low watermark, reads resume.

<Warning title="Backpressure is not only a Netty switch">
`autoRead(false)` protects one channel from pulling more bytes, but it does not tell your business producer, Kafka consumer, scheduler, or WebSocket broadcaster to slow down. Production backpressure must travel to the place where new work is created.
</Warning>

<RuntimeTrace title="Pressure control loop">
Produce | Application writes faster than the peer can receive.
Buffer | Netty queues outbound bytes for the channel.
High watermark | `isWritable()` becomes false.
Pause | The handler stops reading or producing optional work.
Drain | The peer catches up and queued bytes fall below the low watermark.
Resume | `isWritable()` becomes true and input can continue.
</RuntimeTrace>

<RecallPrompt question="why is an unbounded queue not backpressure?">
An unbounded queue delays the overload signal until memory or latency fails. Backpressure deliberately exposes capacity limits early enough to pause, drop, shed, or fail fast.
</RecallPrompt>

## Production implications

Backpressure policy is product policy. Decide what can wait, what can be dropped, what must fail fast, and what should close the connection.

## Performance implications

Good backpressure keeps latency bounded. Bad backpressure hides overload until heap, direct memory, or downstream queues collapse.

## Common bugs

- ignoring `isWritable`
- queuing unbounded messages per channel
- pausing reads without a resume path
- retrying faster during overload
- treating all messages as equally important

## Summary

Backpressure is how event-driven systems remain honest. It turns overload from a surprise into a managed state.
