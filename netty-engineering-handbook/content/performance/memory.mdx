<StoryBlock title="Latency Lives In Memory Behavior">
For network servers, memory pressure often shows up as tail latency before it shows up as an obvious crash.
</StoryBlock>

## Intuition

Networking moves bytes continuously. Allocation patterns, direct memory usage, pooling, and object lifetimes shape both throughput and latency.

<Misconception
  soundsLike="If heap usage looks fine, memory is not the performance problem."
  actually="Networking systems also spend direct memory, pooled buffers, outbound queues, allocator arenas, and GC allocation budget."
  whyItMatters="A service can have healthy heap graphs while direct memory, pending writes, or allocation rate creates tail latency."
/>

## Real-world problem

A server has low average latency but frequent spikes. CPU is acceptable. GC logs show promotion pressure from per-request byte arrays and short-lived wrapper objects.

## Mental model

Separate memory into:

- heap objects used by application logic
- direct buffers used for IO
- pooled arenas and caches
- queued outbound messages

Each pool has different failure modes.

The useful question is not "heap or off-heap?" It is **who owns this memory, what queue can hold it, and what signal tells us it is not returning?**

## Runtime behavior

Netty allocators can pool direct buffers. `ByteBuf` reference counts control when memory returns to the pool. If references leak, direct memory pressure can grow while heap looks healthy.

<RuntimeTrace title="A buffer lifetime">
Allocate | A pooled direct buffer is borrowed for inbound or outbound IO.
Slice | Handlers may create views over the same underlying memory.
Retain | A handler takes a new reference only when it needs a longer lifetime.
Transfer | A write or `fireChannelRead` can move ownership to the next stage.
Release | The final owner releases the reference.
Recycle | The allocator can reuse memory after the reference count reaches zero.
</RuntimeTrace>

## Code example

```java
import io.netty.buffer.ByteBuf;
import io.netty.channel.ChannelHandlerContext;

public final class SafeOutboundEncoder {
    public void write(ChannelHandlerContext ctx, Response response) {
        ByteBuf buffer = ctx.alloc().directBuffer(estimateSize(response));
        try {
            encode(response, buffer);
            ctx.write(buffer);
            buffer = null;
        } finally {
            if (buffer != null) {
                buffer.release();
            }
        }
    }

    private int estimateSize(Response response) {
        return 64 + response.payload().length();
    }

    private void encode(Response response, ByteBuf out) {
        out.writeLong(response.requestId());
        out.writeCharSequence(response.payload(), java.nio.charset.StandardCharsets.UTF_8);
    }
}

record Response(long requestId, String payload) {}
```

The handoff to `ctx.write` transfers ownership. The failure path still releases.

<Warning title="Heap graphs can lie by omission">
A process can have healthy Java heap and still fail from direct memory pressure, allocator fragmentation, or unbounded outbound buffers. Watch direct memory and channel pending bytes beside GC.
</Warning>

<Highlight title="Leak detection is a diagnosis tool">
Netty leak detector output means a reference-counted object escaped its release path. Use higher leak detection levels in lower environments or targeted incident reproduction, then fix ownership instead of hiding the warning.
</Highlight>

<RecallPrompt question="why can direct memory pressure hide from heap-only dashboards?">
Direct buffers and allocator pools live outside the Java heap, so heap usage can look stable while the process runs out of native memory or queues too many outbound bytes.
</RecallPrompt>

## Production implications

Track heap, direct memory, allocation rate, leak detector output, write buffer size, and GC pauses. One graph rarely explains memory behavior by itself.

## Performance implications

Pooling improves steady-state performance when reuse is high. It can hurt if object lifetimes are unclear or if leaks are ignored.

## Common bugs

- converting `ByteBuf` to `byte[]` too early
- retaining buffers across async boundaries without release
- watching heap while direct memory grows
- oversized per-channel queues
- disabling leak detection everywhere

## Summary

Memory discipline is protocol discipline. Know who owns bytes, how long they live, and where pressure is visible.
