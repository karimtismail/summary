<StoryBlock title="Knobs Are The Last Step">
The fastest way to make a system worse is to tune a setting before proving which limit you are hitting.
</StoryBlock>

## Intuition

Tuning starts with a question: what resource or queue is limiting the system? Only then do configuration changes have meaning.

## Real-world problem

A team raises worker threads, socket buffers, and timeouts after a latency incident. Throughput improves briefly, then tail latency gets worse because the real issue was downstream saturation.

## Mental model

Use a loop:

1. define the workload
2. measure baseline
3. identify the bottleneck
4. change one variable
5. compare distribution, not just average
6. keep or revert

## Runtime behavior

Netty exposes many knobs: EventLoop counts, allocator settings, write watermarks, socket options, native transports, batching, and flush behavior. Each knob changes queueing somewhere.

## Code example

```java
bootstrap
    .option(ChannelOption.SO_BACKLOG, 1024)
    .childOption(ChannelOption.TCP_NODELAY, true)
    .childOption(ChannelOption.WRITE_BUFFER_WATER_MARK,
        new WriteBufferWaterMark(32 * 1024, 128 * 1024));
```

These values are not universal. They express a workload hypothesis.

## Production implications

Keep tuning decisions documented with the measurement that justified them. Future teams need to know whether a knob is essential, historical, or accidental.

## Performance implications

Measure p50, p95, p99, max, allocation rate, EventLoop delay, queue depth, retransmits, and downstream latency. A change that improves throughput can harm tail latency.

## Common bugs

- tuning without a reproducible workload
- changing multiple variables at once
- optimizing average latency only
- copying settings from unrelated services
- ignoring downstream bottlenecks

## Summary

Tuning is disciplined evidence collection. Knobs matter only after the bottleneck is understood.
