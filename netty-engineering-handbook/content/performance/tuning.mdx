<StoryBlock title="Knobs Are The Last Step">
The fastest way to make a system worse is to tune a setting before proving which limit you are hitting.
</StoryBlock>

## Intuition

Tuning starts with a question: what resource or queue is limiting the system? Only then do configuration changes have meaning.

## Real-world problem

A team raises worker threads, socket buffers, and timeouts after a latency incident. Throughput improves briefly, then tail latency gets worse because the real issue was downstream saturation.

## Mental model

Use a loop:

1. define the workload
2. measure baseline
3. identify the bottleneck
4. change one variable
5. compare distribution, not just average
6. keep or revert

The most useful tuning question is boring and precise: **which queue got longer first?** If you cannot name the first queue, the knob is probably a guess.

## Visual explanation

<FlowDiagram title="Tuning loop">
Baseline -> Hypothesis -> One change -> Load test -> Compare percentiles -> Decide
</FlowDiagram>

<RuntimeTrace title="Evidence before knobs">
Workload | Define request mix, payload sizes, connection count, and arrival pattern.
Baseline | Record latency percentiles, throughput, errors, CPU, memory, and downstream timing.
Boundary | Locate whether pressure starts at accept, decode, handler, allocator, write buffer, or dependency.
Change | Adjust one variable with an expected direction.
Compare | Check p95 and p99, not only average or total throughput.
Decision | Keep the change only if the original bottleneck moved in the expected way.
</RuntimeTrace>

## Runtime behavior

Netty exposes many knobs: EventLoop counts, allocator settings, write watermarks, socket options, native transports, batching, and flush behavior. Each knob changes queueing somewhere.

Modern Netty documentation may show newer transport setup APIs such as `MultiThreadIoEventLoopGroup` with a transport-specific IO handler factory. Older examples often use `NioEventLoopGroup`. Treat constructor names as version-sensitive; treat the measurement workflow as the stable lesson.

## Code example

```java
import io.netty.bootstrap.ServerBootstrap;
import io.netty.channel.ChannelOption;
import io.netty.channel.WriteBufferWaterMark;

public final class MeasuredServerTuning {
    public static void apply(ServerBootstrap bootstrap) {
        bootstrap
            .option(ChannelOption.SO_BACKLOG, 1024)
            .childOption(ChannelOption.TCP_NODELAY, true)
            .childOption(
                ChannelOption.WRITE_BUFFER_WATER_MARK,
                new WriteBufferWaterMark(32 * 1024, 128 * 1024)
            );
    }
}
```

These values are not universal. They express a workload hypothesis.

<Warning title="A copied knob is technical debt">
`SO_BACKLOG`, `TCP_NODELAY`, watermarks, allocator settings, and native transports are not badges of seniority. Each one should have a benchmark, dashboard, or incident note that explains why it exists.
</Warning>

<Highlight title="What the setting is trying to change">
`SO_BACKLOG` changes the accept queue behavior under connection bursts. `TCP_NODELAY` changes packet coalescing latency tradeoffs. `WRITE_BUFFER_WATER_MARK` changes when a channel becomes non-writable. None of them fixes slow business code.
</Highlight>

## Production implications

Keep tuning decisions documented with the measurement that justified them. Future teams need to know whether a knob is essential, historical, or accidental.

## Performance implications

Measure p50, p95, p99, max, allocation rate, EventLoop delay, queue depth, retransmits, and downstream latency. A change that improves throughput can harm tail latency.

## Common bugs

- tuning without a reproducible workload
- changing multiple variables at once
- optimizing average latency only
- copying settings from unrelated services
- ignoring downstream bottlenecks

## Summary

Tuning is disciplined evidence collection. Knobs matter only after the bottleneck is understood.
