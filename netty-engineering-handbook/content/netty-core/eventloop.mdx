<StoryBlock title="One Lane Per Channel">
Think of a Netty channel as a conversation assigned to one fast lane. The lane is not private forever; one EventLoop usually serves many channels. What it gives you is ordering.

That is the trade: Netty removes a lot of locking pressure by keeping channel work on the same lane, then asks you to keep that lane moving.
</StoryBlock>

## Intuition

An `EventLoop` runs IO events, scheduled tasks, and pipeline callbacks for channels registered to it. For a single channel, those callbacks are ordered by the EventLoop, so handler code can often reason about connection state without defensive locking.

The important word is **often**. The moment you share mutable objects across channels, jump to another executor, or retain work for later, you have to make the ownership and thread boundary explicit again.

<Misconception
  soundsLike="Netty gives every connection its own thread."
  actually="Each channel is assigned to one EventLoop, and one EventLoop can serve many channels."
  whyItMatters="Blocking one callback can delay every other channel assigned to the same EventLoop."
/>

## Real-world problem

A chat server receives a message, validates it, writes it to storage, then broadcasts it. If the storage call blocks inside `channelRead`, the EventLoop cannot keep selecting sockets, running decoders, firing timeouts, or flushing writes for the other channels it owns.

The bug does not look like "my code blocked one user." It looks like random latency across unrelated users.

## Mental model

Treat the EventLoop like a small, fast dispatcher:

- do socket and protocol work on it
- move blocking or CPU-heavy work off it
- return to it when touching the channel again
- never wait on it for work that needs the same loop to complete

The clean question is: **does this callback finish fast enough that I would be comfortable putting hundreds of channels behind it?**

## Runtime behavior

The EventLoop repeatedly checks for IO readiness, executes queued tasks, runs scheduled work, and flushes outbound bytes. Listener callbacks, handler methods, and scheduled tasks often run on that same loop.

<RuntimeTrace title="One EventLoop cycle">
Select | Ask the transport which registered channels are ready.
Read | Pull bytes or readiness events from the channel.
Fire inbound | Move the event through the channel pipeline.
Drain tasks | Run tasks submitted from application threads.
Run timers | Execute scheduled timeouts or delayed work.
Flush | Push queued outbound bytes toward the socket.
</RuntimeTrace>

That cycle is why the central rule is simple: **do not block the EventLoop**.

## Code example

```java
import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.SimpleChannelInboundHandler;
import java.util.concurrent.CompletableFuture;
import java.util.concurrent.Executor;

public final class AccountLookupHandler extends SimpleChannelInboundHandler<AccountRequest> {
    private final Executor blockingExecutor;
    private final AccountClient accountClient;

    public AccountLookupHandler(Executor blockingExecutor, AccountClient accountClient) {
        this.blockingExecutor = blockingExecutor;
        this.accountClient = accountClient;
    }

    @Override
    protected void channelRead0(ChannelHandlerContext ctx, AccountRequest request) {
        CompletableFuture
            .supplyAsync(() -> accountClient.load(request.accountId()), blockingExecutor)
            .whenComplete((account, error) -> ctx.executor().execute(() -> {
                if (!ctx.channel().isActive()) {
                    return;
                }
                if (error != null) {
                    ctx.fireExceptionCaught(error);
                    return;
                }
                ctx.writeAndFlush(new AccountResponse(account.id(), account.name()));
            }));
    }

    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        ctx.close();
    }
}

record AccountRequest(String accountId) {}
record Account(String id, String name) {}
record AccountResponse(String id, String name) {}

interface AccountClient {
    Account load(String accountId);
}
```

The slow lookup runs on a bounded executor. The final channel operation returns to `ctx.executor()`, which is the channel's EventLoop. That keeps Netty's ordering rule intact without parking the IO lane.

<Warning title="Do not smuggle ByteBuf ownership across threads">
If the inbound message contains a reference-counted `ByteBuf`, copy or retain the bytes intentionally before offloading. `SimpleChannelInboundHandler` can auto-release consumed messages after `channelRead0` returns, so a background task must not keep an unowned buffer reference.
</Warning>

<Highlight title="Version note">
Recent Netty documentation may show newer bootstrap APIs such as `MultiThreadIoEventLoopGroup` with a transport-specific IO handler factory. Constructor names can evolve; the durable rule is the same: channel callbacks run on EventLoop lanes, and blocking those lanes delays unrelated channels.
</Highlight>

<RecallPrompt question="what else is delayed when a handler blocks the EventLoop?">
Every channel assigned to that same EventLoop can wait behind the blocked callback: reads, writes, timeouts, listeners, and tasks queued from other threads.
</RecallPrompt>

## Production implications

Look for EventLoop starvation before changing random tuning knobs. Useful evidence includes long stack traces on `nioEventLoop` or `reactor-http-nio` threads, growing write backlog, delayed timeouts, high pending task counts, and latency spikes across unrelated connections.

In Spring Boot WebFlux, Reactor Netty sits under the HTTP server by default. That means the same EventLoop rule applies even if your code is written as controllers and `Mono` or `Flux` pipelines instead of raw Netty handlers.

## Performance implications

The EventLoop model avoids many locks and context switches, but it concentrates latency risk. Keep callbacks bounded, allocation-light, and non-blocking. More EventLoops can help only after the work is already non-blocking and the bottleneck is actual IO parallelism.

## Common bugs

- calling JDBC, JPA, filesystem, or remote blocking clients inside a callback
- calling `sync()`, `await()`, or `block()` on an EventLoop thread
- scheduling unbounded work back onto the EventLoop
- increasing worker thread count to hide blocking code
- sharing mutable handler state across channels without a clear safety rule

## Summary

The EventLoop is the runtime promise that makes Netty feel predictable: a channel's IO work moves through one ordered lane. Keep the lane fast, and Netty scales. Block it, and one slow decision becomes many slow connections.
