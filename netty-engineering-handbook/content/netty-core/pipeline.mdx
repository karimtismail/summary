<StoryBlock title="Protocol Work Has Direction">
Inbound bytes become application messages. Outbound application messages become bytes. The pipeline makes that journey visible.

Good Netty code is not a heroic callback. It is a sequence of small, named transformations that explain the protocol while it runs.
</StoryBlock>

## Intuition

A `ChannelPipeline` is an ordered chain of handlers attached to a channel. Inbound events move from the transport side toward the application side. Outbound events move back toward the transport side.

That direction matters. Handler order is not cosmetic; it defines which layer sees raw bytes, decoded frames, domain messages, encoded responses, and exceptions.

<Misconception
  soundsLike="A pipeline is just a list of callbacks, so I can add handlers wherever the code compiles."
  actually="Handler position changes protocol behavior because inbound and outbound events move in different directions."
  whyItMatters="A decoder before a frame boundary, or an encoder after the write has already passed it, creates bugs that look like random network failure."
/>

## Real-world problem

A production service needs TLS, framing, decoding, authentication, routing, encoding, and error handling. If all of that sits in one handler, every bug feels like "the network is broken."

The pipeline lets you point to the exact stage: "the frame decoder rejected the payload," "auth closed the channel," or "the encoder failed while writing the response."

## Mental model

Read the pipeline in two directions:

- inbound: **bytes to meaning**
- outbound: **meaning to bytes**

Each handler should answer one question:

- what input type do I expect?
- what output type do I emit?
- what error do I own?
- do I consume or pass the message?

## Runtime behavior

Inbound methods such as `channelRead` usually move from the head of the pipeline toward the tail. Outbound operations such as `write` and `flush` move toward the head. When you call `ctx.write(...)`, Netty starts outbound traversal from that handler's context, not from the end of the pipeline.

<RuntimeTrace title="Request through a custom protocol pipeline">
Read bytes | The transport fires an inbound read into the pipeline.
Frame | The frame decoder waits until one complete message boundary exists.
Decode | The message decoder converts bytes into a domain command.
Authorize | The auth handler accepts, rejects, or closes.
Handle | Application logic chooses a response.
Encode | Outbound handlers turn the response back into bytes.
</RuntimeTrace>

This is why many teams name handlers by protocol stage. Names make traces and logs teach the pipeline back to you during an incident.

## Code example

```java
protected void initChannel(SocketChannel ch) {
    ChannelPipeline pipeline = ch.pipeline();

    pipeline.addLast("frameDecoder",
        new LengthFieldBasedFrameDecoder(1 << 20, 0, 4, 0, 4));
    pipeline.addLast("messageDecoder", new MessageDecoder());
    pipeline.addLast("messageEncoder", new MessageEncoder());
    pipeline.addLast("authGate", new AuthGate());
    pipeline.addLast("application", new ApplicationHandler());
}
```

Framing comes before message decoding because the decoder needs complete frames. Encoding is present before the application writes responses because outbound traversal must pass an encoder before bytes reach the socket.

<RecallPrompt question="why can handler order change correctness even when every handler is individually correct?">
Because the pipeline is the protocol path. A handler may require a message to be framed, decoded, authenticated, encoded, or still reference-counted before it can safely do its job.
</RecallPrompt>

## Production implications

Make pipeline composition observable. Log handler names in protocol errors, add counters at semantic boundaries, and keep version-specific handlers isolated. During incidents, you want logs to say "decode failed" or "auth rejected" instead of "channel closed."

If you build on Spring Boot WebFlux, Reactor Netty owns the HTTP pipeline for you. You usually customize at the Reactor Netty or Spring WebFlux layer, but the same concept still explains the runtime: bytes become HTTP requests, requests become handler invocations, responses become outbound bytes.

## Performance implications

Every handler boundary has cost, but clear boundaries usually make performance work easier because bottlenecks become measurable. Avoid handlers that allocate, copy, parse, or transform without adding semantic value.

The best pipeline is not the shortest one. It is the one where each stage has a reason and the hot path stays easy to measure.

## Common bugs

- putting an application handler before the decoder that creates its message type
- writing with `ctx.write` from a context that skips the encoder you expected
- swallowing exceptions without closing, responding, or forwarding
- sharing non-sharable handlers across channels
- mixing blocking business work into protocol handlers
- firing an inbound message twice, or consuming it without release

## Summary

The pipeline is the executable map of your protocol. Design it as a readable path from bytes to meaning and back again, and production behavior becomes much easier to explain.
