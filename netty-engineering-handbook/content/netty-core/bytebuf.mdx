<StoryBlock title="Memory Is Part Of The Protocol">
In high-throughput servers, bytes are not passive data. They are allocated, pooled, sliced, retained, and released.

The useful analogy is ownership, not possession forever. A handler may receive a reference briefly, pass it onward, retain it for later, or release it. Those choices are part of the protocol.
</StoryBlock>

## Intuition

Netty's `ByteBuf` improves on `ByteBuffer` with separate reader and writer indexes, pooling support, direct memory, slicing, composite buffers, and reference counting.

That power comes with a rule the JVM will not enforce for you: if you take ownership of a reference-counted buffer, you must release it exactly when your ownership ends.

<Misconception
  soundsLike="A slice is a safe copy of the bytes."
  actually="A slice usually shares the same underlying memory. `retainedSlice` shares memory and takes a new reference. `copy` creates independent storage."
  whyItMatters="If the parent buffer is released while a shared slice is still used, the later failure looks like corrupted parsing even though the parser was correct."

## Real-world problem

A decoder receives a buffer, slices a frame, stores it for a later async write, and returns. Later the frame fails with an access error or contains unexpected bytes.

The decoding math may be perfect. The ownership story is broken.

## Mental model

Ask two questions every time a `ByteBuf` crosses a handler boundary:

- who owns this reference now?
- when is that owner responsible for `release()`?

Then ask one more question before storing it: **do I need independent bytes, or just a retained view?**

## Runtime behavior

`ByteBuf` has separate reader and writer indexes. Reads such as `readInt()` advance the reader index. Peeks such as `getInt(...)` do not. Reference-counted buffers are deallocated when their count reaches zero.

Derived buffers are the part that surprises people. A view can share the same underlying memory while having its own indexes.

<RuntimeTrace title="Derived buffer decision">
readBytes | Copies bytes into a new buffer and advances the reader index.
getBytes | Copies bytes without advancing the reader index.
slice | Shares memory and does not take new reference ownership.
retainedSlice | Shares memory and increments ownership for the new reference.
copy | Creates independent memory with independent lifetime.
</RuntimeTrace>

Leaks usually appear under load because rare error paths skip release.

## Code example

```java
import io.netty.buffer.ByteBuf;
import io.netty.buffer.DefaultByteBufHolder;
import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.SimpleChannelInboundHandler;
import io.netty.handler.codec.ByteToMessageDecoder;
import java.nio.charset.StandardCharsets;
import java.util.List;

public final class LengthPrefixedFrameDecoder extends ByteToMessageDecoder {
    private static final int HEADER_BYTES = 4;
    private static final int MAX_FRAME_BYTES = 1 << 20;

    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) {
        if (in.readableBytes() < HEADER_BYTES) {
            return;
        }

        int length = in.getInt(in.readerIndex());
        if (length < 0 || length > MAX_FRAME_BYTES) {
            ctx.close();
            return;
        }
        if (in.readableBytes() < HEADER_BYTES + length) {
            return;
        }

        in.skipBytes(HEADER_BYTES);
        ByteBuf payload = in.readRetainedSlice(length);

        try {
            out.add(new Frame(payload));
        } catch (Throwable error) {
            payload.release();
            throw error;
        }
    }
}

final class Frame extends DefaultByteBufHolder {
    public Frame(ByteBuf content) {
        super(content);
    }

    @Override
    public Frame replace(ByteBuf content) {
        return new Frame(content);
    }

    public String utf8Body() {
        return content().toString(StandardCharsets.UTF_8);
    }
}

final class FrameConsumer extends SimpleChannelInboundHandler<Frame> {
    @Override
    protected void channelRead0(ChannelHandlerContext ctx, Frame frame) {
        String body = frame.utf8Body();
        ctx.writeAndFlush("received " + body);
    }
}
```

`readRetainedSlice` gives the emitted `Frame` its own reference. `Frame` extends `DefaultByteBufHolder`, so downstream handlers and Netty utilities can treat it as a reference-counted message. `FrameConsumer` consumes it with `SimpleChannelInboundHandler`, which releases the frame after `channelRead0` returns.

<Warning title="Complete ownership or complete copy">
If a frame will be stored in a queue, retried later, or passed to code that does not understand Netty reference counting, choose deliberately: retain and release on every path, or copy into independent memory. Ambiguous ownership is the leak.
</Warning>

<RecallPrompt question="which operation changes ownership: `slice`, `retainedSlice`, or `copy`?">
`slice` shares memory without taking a new reference. `retainedSlice` shares memory and takes a reference that must be released. `copy` creates independent memory with a separate lifetime.
</RecallPrompt>

## Production implications

Enable leak detection in lower environments and during incident reproduction. Treat leak reports as ownership bugs, not logging noise.

Watch direct memory, allocator metrics, suspicious buffer growth, and decode error paths together. Heap graphs alone can miss the problem because high-performance Netty buffers often live outside the Java heap.

## Performance implications

Pooling, direct memory, and slicing reduce copying and allocation pressure. They also move mistakes out of the comfortable world of garbage collection. Use copies when independence is more important than saving a copy, and use retained views when you can prove the lifetime.

The fastest buffer strategy is the one that is correct first and measured second.

## Common bugs

- using a sliced buffer after the parent reference was released
- retaining without releasing on every path
- releasing a buffer after transferring ownership to the next handler
- converting every message to `byte[]` for convenience
- ignoring leak detector warnings because the app "still works"
- storing pooled buffers in queues without clear lifetime rules

## Summary

`ByteBuf` is central to Netty performance. Correctness depends on clear ownership, deliberate slicing, and disciplined release behavior.
