<StoryBlock title="A Decoder Is A State Machine">
Decoding is not a one-shot parse. It is a careful state transition across partial input.
</StoryBlock>

## Intuition

Decoders must be patient. If there are not enough bytes, they return and wait for more. If there are enough bytes, they consume exactly one message boundary and emit a typed object.

<Misconception
  soundsLike="A decoder is just a parser that runs whenever bytes arrive."
  actually="A decoder is a state machine over partial input. It must peek, validate, wait, consume, and emit without corrupting the reader index."
  whyItMatters="One speculative read in the wrong place can desynchronize every later frame on the connection."
/>

## Real-world problem

A decoder reads a length field, discovers the payload is incomplete, and forgets to reset the reader index. The next read starts in the wrong place and every future frame is corrupt.

## Mental model

Decode only when you can complete the transition. Mark reader state before speculative reads, or use Netty decoders that manage cumulation carefully.

## Runtime behavior

Netty's `ByteToMessageDecoder` accumulates inbound bytes and calls `decode` repeatedly while progress is possible. If your decoder consumes bytes without emitting or waiting correctly, it can spin or corrupt data.

<RuntimeTrace title="Safe decoder cycle">
Peek | Inspect length or version without permanently moving the reader index.
Validate | Reject impossible lengths before allocating or retaining.
Wait | Return if the full frame has not arrived.
Consume | Advance indexes only after the complete frame is present.
Emit | Pass one typed message with a clear ownership rule.
Repeat | Let Netty call `decode` again if another complete frame is buffered.
</RuntimeTrace>

## Code example

```java
import io.netty.buffer.ByteBuf;
import io.netty.channel.ChannelHandlerContext;
import io.netty.handler.codec.ByteToMessageDecoder;
import io.netty.handler.codec.CorruptedFrameException;
import java.util.List;

public final class CommandDecoder extends ByteToMessageDecoder {
    private static final int HEADER_BYTES = 5; // version + length
    private static final int MAX_PAYLOAD_BYTES = 1 << 20;

    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) throws Exception {
        if (in.readableBytes() < HEADER_BYTES) {
            return;
        }

        int start = in.readerIndex();
        byte version = in.getByte(start);
        int length = in.getInt(start + 1);

        if (version != 1) {
            throw new CorruptedFrameException("unsupported protocol version: " + version);
        }
        if (length < 0 || length > MAX_PAYLOAD_BYTES) {
            throw new CorruptedFrameException("invalid frame length: " + length);
        }
        if (in.readableBytes() < HEADER_BYTES + length) {
            return;
        }

        in.skipBytes(HEADER_BYTES);
        out.add(new Command(version, in.readRetainedSlice(length)));
    }
}

record Command(byte version, ByteBuf payload) {
    void release() {
        payload.release();
    }
}
```

The decoder peeks with `getByte` and `getInt`, so partial input does not disturb the reader index. It advances only when the whole frame is present, validates impossible lengths before retaining memory, and emits a command with an explicit release path.

<Warning title="Never trust the length field">
A malicious or broken peer can claim a huge payload. Always set a maximum frame size before allocating, retaining, or waiting forever for bytes that should not be accepted.
</Warning>

<RecallPrompt question="why peek before consuming header bytes?">
Peeking lets the decoder inspect version and length while preserving the reader index if the full frame has not arrived yet.
</RecallPrompt>

## Production implications

Decoder failures should include protocol version, remote address, frame size, and reason. Do not log raw payloads unless privacy and volume are controlled.

## Performance implications

Avoid copying payloads unless the next layer needs independent ownership. Use retained slices carefully and release them downstream.

## Common bugs

- advancing the reader index before a complete frame exists
- emitting mutable buffers without ownership rules
- catching decode errors and continuing with corrupt state
- allocating objects during speculative parses
- missing protocol version checks

## Summary

Decoders are state machines under backpressure and partial input. Correctness comes from consuming only what you can prove is complete.
