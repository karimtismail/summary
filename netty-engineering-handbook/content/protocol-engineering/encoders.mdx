<StoryBlock title="The Wire Format Is A Promise">
Once clients depend on your bytes, an encoder is not just implementation. It is compatibility policy.
</StoryBlock>

## Intuition

Encoding is where application meaning becomes wire format. The encoder decides field order, byte order, lengths, version markers, and optional data.

Good encoders are boring because their output is predictable.

## Real-world problem

A service adds a field to a binary response without a version marker or length-delimited extension area. Older clients misread the rest of the frame.

## Mental model

Treat the encoder as a serializer plus a compatibility boundary. It should know the protocol version and produce frames older clients can either understand or reject cleanly.

## Runtime behavior

Netty encoders run outbound. They often allocate an output `ByteBuf`, write header fields, serialize payloads, then pass bytes toward the transport.

<RuntimeTrace title="Stable encoder cycle">
Choose version | Decide the wire contract this client should receive.
Reserve header | Leave room for length, flags, or checksum fields.
Write payload | Encode fields in a deterministic order and charset.
Patch metadata | Fill length or checksum after payload size is known.
Pass bytes | Let framing, TLS, and transport handlers continue outbound.
</RuntimeTrace>

## Code example

```java
import io.netty.buffer.ByteBuf;
import io.netty.channel.ChannelHandlerContext;
import io.netty.handler.codec.MessageToByteEncoder;
import java.nio.charset.StandardCharsets;

public final class ResponseEncoder extends MessageToByteEncoder<Response> {
    @Override
    protected void encode(ChannelHandlerContext ctx, Response msg, ByteBuf out) {
        int frameStart = out.writerIndex();

        out.writeByte(msg.version());
        out.writeInt(0); // payload length placeholder
        out.writeLong(msg.requestId());
        out.writeCharSequence(msg.payload(), StandardCharsets.UTF_8);

        int payloadLength = out.writerIndex() - frameStart - 1 - Integer.BYTES;
        out.setInt(frameStart + 1, payloadLength);
    }
}

record Response(byte version, long requestId, String payload) {}
```

Reserving the length lets the encoder fill it after the payload is known.

<Warning title="Encoding is part of compatibility">
If payload text, byte order, length calculation, or field order changes, old clients may parse valid bytes as the wrong message. Lock the wire contract with golden byte tests before changing it.
</Warning>

## Production implications

Capture wire-format compatibility in tests. Golden byte tests are useful because they detect accidental changes to protocol output.

## Performance implications

Encoding can become allocation-heavy. Prefer writing into provided buffers, sizing buffers reasonably, and avoiding intermediate strings or arrays.

## Common bugs

- changing field order without a version plan
- using platform-default character encodings
- forgetting byte order
- writing a length before knowing the final payload
- allocating temporary arrays for every field

## Summary

Encoders are compatibility machinery. Make the wire format explicit, versioned, and tested.
