<StoryBlock title="HTTP Is A Protocol Family">
HTTP is not just text over TCP. It is a set of semantics around requests, responses, headers, caching, intermediaries, and connection reuse.
</StoryBlock>

## Intuition

HTTP gives applications a common language: methods, paths, status codes, headers, bodies, and upgrade mechanisms. Netty gives you the tools to decide how much of that protocol you want to handle directly.

<Misconception
  soundsLike="HTTP is simple enough to parse as strings for a small server."
  actually="HTTP includes body framing, chunking, connection reuse, header rules, upgrades, intermediaries, and malformed-input behavior."
  whyItMatters="String parsing often works in demos and then fails under chunked bodies, persistent connections, proxies, or malicious requests."
/>

## Real-world problem

A custom server treats the first blank line as the end of an HTTP request. That shortcut fails once the server meets chunked bodies, persistent connections, compression, or malformed headers. The problem is trying to parse a mature protocol as a string exercise.

## Mental model

Let protocol codecs handle syntax. Put application code behind complete HTTP messages or streamed body chunks, depending on the workload.

HTTP is two contracts at once:

- semantic contract: method, path, headers, status, caching, authentication
- transport contract: connection reuse, body streaming, backpressure, timeouts, intermediaries

## Runtime behavior

HTTP/1.1 can reuse connections. Requests and responses must preserve ordering on a connection unless pipelining rules are deliberately handled. Large bodies may arrive as chunks.

<RuntimeTrace title="HTTP request lifecycle">
Decode start line | Method, URI, and protocol version are parsed.
Decode headers | Host, content length, transfer encoding, auth, and routing metadata appear.
Handle body | The body is aggregated or streamed as chunks.
Apply semantics | The application validates method, path, auth, content type, and size.
Write response | Status, headers, and body are emitted in protocol order.
Keep or close | Connection policy follows keep-alive, errors, idle timeout, or shutdown.
</RuntimeTrace>

## Code example

```java
import io.netty.buffer.Unpooled;
import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.ChannelPipeline;
import io.netty.channel.SimpleChannelInboundHandler;
import io.netty.handler.codec.http.DefaultFullHttpResponse;
import io.netty.handler.codec.http.FullHttpRequest;
import io.netty.handler.codec.http.FullHttpResponse;
import io.netty.handler.codec.http.HttpHeaderNames;
import io.netty.handler.codec.http.HttpObjectAggregator;
import io.netty.handler.codec.http.HttpResponseStatus;
import io.netty.handler.codec.http.HttpServerCodec;
import java.nio.charset.StandardCharsets;

public final class SmallHttpPipeline {
    public static void configure(ChannelPipeline pipeline) {
pipeline.addLast(new HttpServerCodec());
pipeline.addLast(new HttpObjectAggregator(1 << 20));
        pipeline.addLast(new HealthHandler());
    }
}

final class HealthHandler extends SimpleChannelInboundHandler<FullHttpRequest> {
    @Override
    protected void channelRead0(ChannelHandlerContext ctx, FullHttpRequest request) {
        FullHttpResponse response = new DefaultFullHttpResponse(
            request.protocolVersion(),
            HttpResponseStatus.OK,
            Unpooled.copiedBuffer("ok", StandardCharsets.UTF_8)
        );
        response.headers().set(HttpHeaderNames.CONTENT_LENGTH, response.content().readableBytes());
        ctx.writeAndFlush(response);
    }
}
```

Aggregation is convenient, but it must be bounded.

<Warning title="HTTP parsing is security-sensitive">
Do not hand-roll header parsing, body limits, chunk handling, or upgrade behavior unless you are deliberately writing protocol infrastructure. Mature codecs exist because real HTTP has ambiguous-looking edge cases.
</Warning>

<RecallPrompt question="when is `HttpObjectAggregator` a good fit?">
It is a good fit for bounded, small request or response bodies where holding the full message in memory is acceptable and the maximum size is explicit.
</RecallPrompt>

## Production implications

Validate headers, body sizes, method support, idle timeouts, and connection reuse behavior. Intermediaries may change how clients reach your server.

## Performance implications

Aggregating bodies is simple and memory-expensive. Streaming is more complex and often required for large uploads, downloads, and proxy workloads.

## Common bugs

- aggregating unbounded request bodies
- ignoring keep-alive semantics
- parsing HTTP manually
- blocking while streaming a body
- losing request context during async writes

## Summary

HTTP is a rich application protocol. Use Netty's codecs for syntax and reserve custom code for application semantics.
