<StoryBlock title="The Bridge Out Of Request-Response">
WebSocket belongs in protocol engineering because it changes the shape of an HTTP connection. The dedicated WebSocket track goes deeper; this chapter gives the map.
</StoryBlock>

## Intuition

WebSocket is useful when both client and server need to send messages independently over one connection. It is common for collaboration, telemetry, market data, games, and live control planes.

At this level, the main idea is the protocol transition: an HTTP request asks for an upgrade, the server accepts or rejects it, and accepted connections stop behaving like short request-response exchanges.

## Real-world problem

A product polls every second for updates. Under load, most requests return no change and still pay HTTP overhead. WebSocket can move the system from repeated request-response to event delivery.

## Mental model

WebSocket has two phases:

- HTTP upgrade handshake
- framed bidirectional messages

Use this chapter as the bridge. For the detailed mechanics, continue to [Handshake and Upgrade](/handbook/websocket/handshake-upgrade), [Frames and Lifecycle](/handbook/websocket/frames-lifecycle), and [Server Design](/handbook/websocket/server-design).

## Runtime behavior

Netty handles the HTTP parsing and WebSocket upgrade with protocol handlers. After a successful upgrade, later handlers receive WebSocket frames instead of normal HTTP requests.

<RuntimeTrace title="Upgrade boundary">
HTTP parse | The request path and headers are decoded as HTTP.
Validate | The server checks upgrade headers, auth, origin, and path policy.
Switch | A 101 response changes the protocol on the same connection.
Frame stream | Text, binary, ping, pong, and close frames move through the pipeline.
Delegate | Application-specific routing, heartbeat, and scaling belong to the WebSocket track.
</RuntimeTrace>

## Code example

```java
import io.netty.channel.ChannelHandler;
import io.netty.channel.ChannelPipeline;
import io.netty.handler.codec.http.HttpObjectAggregator;
import io.netty.handler.codec.http.HttpServerCodec;
import io.netty.handler.codec.http.websocketx.WebSocketServerProtocolHandler;

public final class WebSocketProtocolBridge {
    public static void configure(ChannelPipeline pipeline, ChannelHandler applicationFrameHandler) {
        pipeline.addLast(new HttpServerCodec());
        pipeline.addLast(new HttpObjectAggregator(8192));
        pipeline.addLast(new WebSocketServerProtocolHandler("/events"));
        pipeline.addLast(applicationFrameHandler);
    }
}
```

The HTTP codec is still needed for the handshake. The application frame handler is deliberately left abstract here because the dedicated WebSocket chapters own routing, heartbeat, backpressure, Spring support, scaling, and security.

<RecallPrompt question="what changes after the WebSocket upgrade?">
The connection stops being ordinary request-response HTTP and becomes a bidirectional framed channel.
</RecallPrompt>

## Production implications

Use the protocol overview to place the boundary correctly. Production details such as idle detection, heartbeat policy, authentication refresh, connection draining, fanout pressure, and registry cleanup are covered in the dedicated WebSocket section.

## Performance implications

The important performance shift is that connection count and per-connection state matter alongside request throughput. Long-lived channels change capacity planning.

## Common bugs

- treating post-upgrade frames like new HTTP requests
- hiding auth, origin, and upgrade policy in ordinary route logic
- forgetting that one accepted connection may live for hours
- skipping the dedicated lifecycle and scaling design after the handshake works

## Summary

WebSocket is the point where HTTP request-response becomes a bidirectional frame stream. This overview teaches the boundary; the WebSocket track teaches the operational design.
